# Connect AI Brain Cleanup Plan v1

## 1. 문서 목적

이 문서는 brain 정리 전 기준점을 만들기 위한 계획 문서다.

목적은 다음과 같다.

- 운영 지식과 테스트 찌꺼기를 구분한다.
- 추후 초기화, 아카이브, 정리 작업의 안전 기준으로 사용한다.
- brain 파일을 바로 건드리지 않고 먼저 사람이 검토할 수 있는 기준을 남긴다.

## 2. 현재 문제

Connect AI는 로컬 brain 폴더에 대화, 결정, 작업 결과가 계속 쌓이는 구조다.

최근 Telegram Mobile Control, Safe Auto, Role Routing, Gemini Native Provider 테스트를 반복하면서 brain 지식 수가 크게 증가했다.

이로 인해 다음 문제가 생길 수 있다.

- 테스트 반복으로 brain 지식 수 증가
- 실패 로그, 임시 판단, 오래된 우선순위가 남을 가능성
- 현재 상태와 과거 테스트 결과가 섞일 가능성
- CEO가 오래된 판단을 할 수 있음

즉, 지금 필요한 것은 즉시 삭제가 아니라, **정리 기준과 절차를 먼저 문서화하는 것**이다.

## 3. 절대 바로 삭제하면 안 되는 이유

brain에는 단순 테스트 기록만 있는 것이 아니라 중요한 결정 근거가 섞여 있을 수 있다.

바로 삭제하면 안 되는 이유는 다음과 같다.

- 중요한 결정 근거가 섞여 있을 수 있다.
- 역할/라우팅/모바일 UX 정책이 brain에 남아 있을 수 있다.
- 삭제 전에 현재 상태 문서와 비교가 필요하다.

따라서 brain 정리는 삭제보다 먼저 **분류와 검토**가 우선이다.

## 4. 보존해야 할 운영 지식

다음 내용은 운영 지식으로 보존해야 한다.

- 공식 역할 9개
- 비서/영숙이 사용자-facing 입구라는 정책
- CEO는 내부 의사결정/작업 분배 역할이라는 정책
- 모델 라우팅 정책 v1
- Telegram Mobile Control v1 구조
- `/go`, `/status`, `/results`, `/stop` 사용 흐름
- Safe Auto 동작 원칙
- Gemini native provider 결정 이유
- Gemini OpenAI-compatible 경로를 쓰지 않기로 한 이유
- 현재 완료된 stable 태그와 주요 기능

이 항목들은 테스트 로그와 분리해서 현재 상태 문서 또는 정책 문서에 남겨야 한다.

## 5. 정리 후보 지식

다음은 정리 후보가 될 수 있는 지식이다.

- 반복 테스트로 생성된 동일한 5060 AI 수익화 강의 결과
- 실패한 Gemini OpenAI-compatible 테스트 로그
- API key invalid 테스트 결과
- 오래된 `/delegate`, `/run-safe`, `/result` 우선순위 판단
- 이미 완료된 기능을 "다음 우선순위"로 말하게 만드는 과거 기록
- 중복된 `run-plan`, `auto-safe`, `cancel-plan` 테스트 기록

이 후보들은 운영 지식인지, 테스트 기록인지 먼저 구분해야 한다.

## 6. 정리 방식 후보

brain 정리 방식은 여러 가지가 가능하다.

- 삭제
- archive 폴더 이동
- `old/test` prefix 부여
- summary 문서로 압축
- current-state 문서로 대체
- 운영 지식과 테스트 기록 분리

현재 단계에서는 삭제보다 분리와 아카이브를 우선 고려해야 한다.

## 7. 추천 절차

brain 정리는 아래 순서로 진행하는 것이 안전하다.

1. brain inventory 확인
2. 현재 상태 문서와 비교
3. 보존/아카이브/삭제 후보 분류
4. 테스트 데이터만 별도 archive
5. current state와 model routing policy를 최신 기준으로 남김
6. `/ceo`와 `/go`로 재검증

이 절차를 먼저 정리해 두면, 실제 삭제 작업을 나중에 안전하게 분리할 수 있다.

## 8. 정리 후 검증 기준

brain 정리 후에는 다음 기준으로 상태를 확인한다.

- `/ceo`가 오래된 우선순위를 말하지 않는지
- `/go`가 현재 정책대로 동작하는지
- 비서/영숙은 로컬 Qwen을 유지하는지
- CEO/현빈/planner는 Gemini를 사용하는지
- `/results` 결과가 현재 목표와 맞는지

이 검증은 정리 작업이 실제 운영 정책을 깨뜨리지 않았는지 확인하기 위한 것이다.

## 9. 주의사항

- brain 파일 직접 삭제는 별도 작업으로 분리한다.
- 삭제 전 백업 또는 archive가 필요하다.
- 자동 정리 스크립트는 아직 만들지 않는다.
- 먼저 사람이 검토 가능한 기준 문서부터 만든다.

## 10. 다음 액션

brain 정리 계획의 다음 단계는 아래와 같다.

1. brain inventory 출력 명령 정리
2. 정리 후보 목록 만들기
3. archive 전략 결정
4. 필요하면 `brain-cleanup-safe` 스크립트 설계

이 문서는 그 이후 실제 정리 작업의 기준점이 된다.
