# Connect AI 운영 테스트 로그

## 1. 문서 목적

이 문서는 Connect AI를 실제 업무 자동화 도구로 쓰면서 발생한 결과, 문제, 개선점을 누적 기록하기 위한 운영 로그다.

- 실사용 테스트 결과를 계속 축적한다.
- 기능 오류, 품질 문제, 라우팅 문제, brain 문제를 구분해서 기록한다.
- 다음 개선 우선순위를 판단할 때 근거로 사용한다.
- 상품화 목적의 이상적인 설계보다, 실제 /go 테스트에서 드러난 문제를 기준으로 반복 개선한다.

이 문서는 완성형 설계 문서가 아니라 운영 중 갱신되는 실험 기록이다.

## 2. 현재 운영 원칙

운영 기준은 아래를 우선한다.

- Writer는 기본적으로 local Qwen을 유지한다.
- CEO, 현빈, planner purpose는 Gemini 우선으로 둔다.
- 비서, 영숙은 local Qwen 고정으로 둔다.
- Writer 품질 개선은 후순위로 둔다.
- 우선순위는 전체 자동화 흐름의 안정성과 실제 업무 적용성 검증이다.

즉, 문체 품질을 먼저 다듬기보다, 작업 지시가 실제로 끝까지 흘러가고 결과가 남는지부터 본다.

## 3. 테스트 기록 양식

각 테스트는 아래 항목으로 기록한다.

- 날짜
- 테스트 목표
- 입력한 /go 명령
- planId
- 생성된 작업 목록
- 완료/실패 여부
- 사용된 모델 라우팅
- 좋았던 점
- 문제점
- 원인 추정
- 개선 후보
- 우선순위
- 후속 액션

### 기록 예시

- 날짜: `2026-05-28`
- 테스트 목표: `5060 AI 수익화 강의 /go 흐름 확인`
- 입력한 /go 명령: `/go 5060 AI 수익화 강의`
- planId: `delegate-20260528031635-x3i7`
- 생성된 작업 목록:
  - Researcher
  - Business
  - Writer
- 완료/실패 여부: `일부 성공, Writer timeout 실패`
- 사용된 모델 라우팅:
  - Researcher: 기본 local Qwen
  - Business: Gemini 우선
  - Writer: local Qwen
- 좋았던 점:
  - /go 흐름 자체는 정상 동작했다.
  - dependency context metadata를 결과에서 확인할 수 있었다.
- 문제점:
  - Writer가 timeout으로 끝났다.
  - downstream dependency context가 길어질 때 실행 시간이 늘어날 수 있다.
- 원인 추정:
  - Writer prompt에 dependency 정보가 중복으로 들어갔을 가능성
  - local Qwen이 60초 안에 처리하기 어려운 context 크기
- 개선 후보:
  - dependency prompt block 압축
  - dependencyEvidence는 prompt에서 제외
  - header/tracker metadata는 유지
- 우선순위: `높음`
- 후속 액션:
  - 실제 업무형 /go 테스트를 더 돌려서 반복 여부 확인

## 4. 문제 분류 기준

문제는 아래 기준으로 분류한다.

- 기능 오류: 명령이 동작하지 않거나 결과가 잘못 저장되는 문제
- timeout: 모델 응답이 제한 시간 안에 끝나지 않는 문제
- 결과 품질 부족: 결과는 생성되지만 실사용에 부족한 문제
- 역할 라우팅 이상: 기대한 모델/역할이 아닌 곳으로 라우팅되는 문제
- dependency 반영 부족: 선행 작업 결과가 downstream에 충분히 반영되지 않는 문제
- brain context 문제: brain 파일이나 context source가 잘못 반영되는 문제
- 모바일 UX 불편: Telegram/모바일 사용 흐름이 불편한 문제
- 비용/속도 문제: 토큰 사용량, 응답 속도, 처리 안정성 문제

이 분류는 원인을 빨리 특정하기 위한 기준이며, 하나의 테스트가 여러 분류에 동시에 걸릴 수 있다.

## 5. 현재까지 확인된 기준 상태

현재까지는 아래 상태로 본다.

- `/go` 정상 동작
- `/status` 정상 동작
- `/results` 정상 동작
- `/result` 정상 동작
- `/queue` / `/live` 정상 확인
- Safe Auto 완료 후 OFF 확인
- dependency context metadata 표시 가능
- Writer는 Qwen으로 유지
- Writer 문체 품질은 후순위 개선 대상

이 상태는 계속 갱신할 수 있다. 실제 테스트에서 깨지면 바로 수정한다.

## 7. 다음 테스트 후보

다음으로 확인할 후보 테스트는 아래와 같다.

- AI 교육 모임 90분 실습형 강의안 준비

이 후보들은 특정 주제에 고정된 테스트가 아니라, 실제 업무 자동화 흐름이 범용적으로 잘 도는지 확인하기 위한 샘플이다.

## 8. 운영 판단 기준

운영 중에는 아래 원칙을 따른다.

- 한 번의 결과 품질 문제로 바로 코드 수정하지 않는다.
- 3~5회 테스트 후 반복되는 문제를 우선 개선한다.
- 기능 실패와 품질 부족을 구분한다.
- 테스트 로그를 근거로 다음 개발 우선순위를 정한다.

이 기준은 불필요한 과잉 수정과 감정적 대응을 줄이기 위한 것이다.

## 9. 결론

이제부터는 기능 추가보다 실사용 테스트와 반복 개선이 중심이다.

Connect AI를 사용자 개인 업무 자동화 도구로 안정화하는 것이 목표다.

이 문서는 그 과정에서 나온 결과와 실패를 축적하는 운영 기준 문서로 사용한다.

## 테스트 기록: /go planner objective drift 수정

### 날짜

* `2026-05-28`

### 테스트 목표

* `/go`가 최근 brain/session 맥락에 끌려가지 않고 현재 objective를 우선하는지 확인
* 5060 AI 수익화 테스트 흐름이 다른 목표에 잘못 섞이지 않는지 확인

### 확인한 /go 명령

```txt
/go 이번 주 AI 교육 모임에서 사용할 90분짜리 실습형 강의안을 준비해줘. 대상은 AI 초보자이고, 목표는 참가자가 수업 안에서 직접 결과물 하나를 만들어보는 것이다.
```

```txt
/go 5060 대상 AI 수익화 강의 사업을 준비해줘. 2주 안에 첫 파일럿 모집글까지 올리고 싶어.
```

```txt
/go 다음 주 고객사 미팅 자료를 1시간 안에 정리해줘.
```

### 결과

* 90분 AI 교육 강의안 목표에서 5060 파일럿 흐름으로 끌려가던 문제가 개선됨
* 5060 AI 수익화 목표는 기존 course-pilot 흐름을 정상 허용함
* 고객사 미팅 자료 목표도 미팅 자료 정리 방향으로 생성됨

### 좋았던 점

* planner objective drift 방지 효과 확인
* 최근 session/brain context가 현재 objective를 덮어쓰는 문제를 줄임
* 정상적인 5060 목표는 계속 허용됨

### 남은 문제

* Writer 결과 품질은 후속 개선 대상
* `/status` 단계 표시와 Safe Auto 완료 카운트 메시지는 별도 UX 개선 후보
* 장기적으로 회사/프로젝트 폴더 분리와 skill/template 저장 방식도 테스트 필요

### 후속 액션

* 실제 업무형 `/go` 테스트를 3~5회 더 누적
* 반복되는 문제만 골라 다음 개선 우선순위로 선정
