# Worker Prompt Quality General v1

## 1. 문서 목적

Researcher / Writer 품질 개선을 특정 주제가 아니라 범용 원칙으로 설계한다.

이 문서는 5060 AI 교육 같은 단일 테스트 예시를 위한 전용 규칙이 아니라, 앞으로 들어올 다양한 업무 유형에 공통 적용 가능한 프롬프트 품질 기준을 정리한다.

## 2. 현재 문제

현재 Researcher / Writer는 다음 문제를 보일 수 있다.
- 사용자의 목표, 대상, 기간, 제약을 충분히 해석하지 못한다.
- AI 관련 요청을 받으면 기술 일반론으로 빠질 가능성이 있다.
- 결과가 실제 실행 가능한 산출물보다 설명문에 가까워진다.
- `/results` 요약만 보면 품질 문제가 더 두드러진다.

이 문제는 특정 주제의 난이도 때문만이 아니라, 역할 프롬프트가 범용 분해 규칙보다 예시성 주제에 끌릴 때 더 잘 드러난다.

## 3. 절대 피해야 할 방향

다음 방향은 피해야 한다.
- 5060 AI 교육 전용 규칙을 `roleRules`에 넣기
- 머신러닝/딥러닝 금지를 모든 작업에 적용하기
- 특정 테스트 과제를 agent goal에 고정하기
- Researcher를 무조건 외부 API로 보내기

이런 방식은 특정 테스트를 잠깐 맞추더라도, 다른 실제 업무가 들어왔을 때 오히려 품질을 떨어뜨린다.

## 4. 개선 원칙

개선은 아래 원칙을 따른다.
- `roleRules`는 범용 원칙만 담는다.
- `task context`가 이번 과제의 대상, 목표, 기간, 제약을 담당한다.
- brain agent goal은 역할의 장기 원칙만 담는다.
- 특정 주제 조건은 task마다 전달되어야 한다.

즉, 역할은 “무엇을 어떻게 생각할지”를 정하고, 과제는 “이번에 무엇을 처리할지”를 정한다.

## 5. Researcher 범용 원칙

Researcher는 조사 대상이 무엇이든 먼저 아래를 수행해야 한다.
- 사용자 목표를 먼저 분해한다.
- 대상 고객, 사용자, 시장, 기간, 제약을 식별한다.
- 필요한 조사 관점을 선택한다.

예시 관점:
- 교육: 수강생 니즈, 구매 장벽, 커리큘럼 적합성, 모집 채널
- 마케팅: 고객 세그먼트, 메시지, 채널, 전환 요인
- 개발: 기술 선택지, 구현 난이도, 리스크, 검증 방법
- 사업: 수익모델, 가격, 운영 리스크, 차별화

추가 원칙:
- 근거 없는 최신 트렌드나 데이터는 단정하지 않는다.
- 외부 사실 확인이 필요한 경우 `확인 필요`로 표시한다.
- 결과는 가설, 근거, 리스크, 다음 검증 질문으로 정리한다.

## 6. Writer 범용 원칙

Writer는 어떤 산출물이든 아래 기준을 먼저 반영해야 한다.
- 대상 독자와 채널을 먼저 반영한다.
- 추상 설명보다 바로 쓸 수 있는 초안을 만든다.
- 요청된 산출물 형식에 맞춘다.

대표 산출물 형식:
- 모집글
- 커리큘럼
- 스크립트
- 제안서
- 상세페이지
- 이메일/DM

추가 원칙:
- 후킹, 본문, CTA, 제목, 설명 등 필요한 구성요소를 포함한다.
- 독자의 이해 수준에 맞게 난이도를 조절한다.
- 과제와 무관한 이론 설명으로 빠지지 않는다.

## 7. self-rag / 자가검증 반영 방향

자가검증 ON은 별도 LLM 검증 호출이 아니라 prompt-level self-rag 구조다.

따라서 다음 파일과 설정이 어떻게 연결되는지 확인해야 한다.
- `verified.md`
- `rag_mode.txt`
- `self_rag_criteria.md`

Researcher / Writer의 자가검증 기준은 다음 중심이어야 한다.
- 사용자 목표 적합성
- 근거
- 실행 가능성
- 산출물 형식 준수

자가검증은 “맞는 말을 했는가”보다 “이번 과제에 실제로 쓸 수 있는 결과인가”를 확인하는 장치여야 한다.

## 8. local Qwen vs external Gemini 판단

기본 정책은 다음과 같다.
- 내부 brain 기반 정리와 구조화는 local Qwen 우선
- 최신 트렌드, 외부 데이터, 사실 확인이 필요한 경우 external 또는 검색 도구 검토
- Researcher를 기본 외부 API로 보내지 않는다
- `researcherModel`을 명시했을 때만 외부 사용하도록 하는 정책은 유지한다

즉, local Qwen은 역할 정의와 산출물 구조를 잡는 데 우선 사용하고, 외부 모델은 최신성이나 사실 확인이 필요한 경우에만 보조적으로 사용한다.

## 9. 수정 후보

개선 후보는 아래와 같다.
- `src/extension.ts`의 `_runSafeBuildSystemPrompt()` roleRules
- `D:\dev\brain\_company\_agents\researcher\goal.md`
- `D:\dev\brain\_company\_agents\writer\goal.md`
- 필요 시 Researcher self-rag 기준 파일
- 필요 시 Writer self-rag 기준 파일

이때 중요한 점은, 역할을 특정 주제에 묶는 것이 아니라 범용 원칙을 강화하는 것이다.

## 10. 테스트 기준

다음 요청들에서 동일한 품질 원칙이 유지되는지 확인한다.
- 5060 AI 수익화 강의 요청
- 40/50 자영업자 마케팅 자동화 요청
- GitHub 연동 작업 요청
- 콘텐츠 자동화 요청

검증 포인트:
- 특정 주제에 고정되지 않는지
- 목표, 대상, 산출물 중심으로 분해되는지
- 실행 가능한 형태로 정리되는지
- 불필요한 기술 일반론으로 빠지지 않는지

## 11. 결론

먼저 범용 설계 문서를 확정한다.
- 이후 `roleRules`를 최소 수정한다.
- 필요하면 brain agent goal을 보강한다.
- 그 다음 `/go` 테스트로 품질을 비교한다.

핵심은 특정 예시를 role level에 고정하는 것이 아니라, 어떤 업무가 와도 `목표 → 대상 → 제약 → 산출물` 순서로 안정적으로 분해·작성되게 만드는 것이다.
